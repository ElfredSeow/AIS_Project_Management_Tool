SECTION 1

Product Overview — AIS Project Tracker

The AIS Project Tracker is an internal project management application built for engineering teams at AIS (Precision Engineering) to track both hardware and software initiatives. It addresses a common pain point in engineering organisations: projects that stagnate without visibility, accountability, or a clear audit trail of status changes.

The app is deployed as a Power Apps Code App — a React/TypeScript application hosted inside Power Platform — giving it seamless integration with Dataverse for data storage and Power Platform's authentication layer for access control.

🎯 Problem Solved

Prevents project stagnancy by surfacing overdue milestones and stagnant statuses with visual alerts so managers can act immediately.

👥 Who Uses It

Project Managers, Developers, End Users, Ops Managers, and Product Owners across hardware and software engineering teams.

🏗️ Where It Lives

Deployed on Power Platform as a Code App backed by Dataverse tables. Authentication is managed by Power Platform.

SECTION 2

Key Features & User Scenarios

Kanban Dashboard

The primary interface is a four-column Kanban board. Projects are grouped by status and each column is colour-coded to give instant visual clarity.

● Ideation ● In Progress ● Completed ● De-Prioritised
Drag & Drop Status Changes

Cards can be dragged between columns. Dropping a card in a different column automatically updates the project's status in Dataverse and shows a confirmation toast. Drop zones highlight with a dashed border when a card is being dragged over them.

Project Cards

Each card shows the project name, type badge (Hardware or Software), a date-status left border (red = overdue, amber = due soon, emerald = on track), and action buttons for Edit, Expand, and View.

Project Detail Modal

Full-screen modal accessible from the "View" button. Displays progress bar, project team grouped by role, problem statement, proposed solution, expected benefits, estimated man-hours saved, timeline, milestones with completion status, and clickable media URL tags.

Key User Scenarios

Scenario How it works
Daily Standup Review Manager opens dashboard, filters to "In Progress" projects, spots amber/red-highlighted stagnant items requiring follow-up
Project Creation Owner clicks "+" in any column or the global create button; fills in the project form with name, problem statement, type, contributors, milestones, and media URLs
Status Update Drag a card to a different column — system updates Dataverse and logs the change automatically
Escalation Discovery Filter to stagnant projects, open detail modal to review full milestone history and last activity
Completion Tracking Mark project as Completed; system records final date and displays total estimated man-hours saved in the KPI summary

Filtering & Sorting

The header bar provides filters by project type (Hardware / Software), sorting options (by name, date, priority), and a light/dark theme toggle. An export dialog allows downloading the project list as Excel or PDF.

All data is stored in Dataverse. The application uses one primary table and one in-memory table for media URLs.

What is PowerCodex?

PowerCodex is an open-source CLI initializer for teams building Power Apps Code Apps. It scaffolds a production-ready starter and wires up a spec-driven development workflow using OpenSpec and GitHub Copilot prompt files — so every idea moves from exploration to reviewable requirements to implementation without losing decisions in chat history.

💡 Core philosophy: Keep intent close to the code. Every idea becomes a file that can be reviewed, changed, implemented, verified, and archived. Decisions live in the repository — not in a chat window.

PowerCodex OPSX Workflow (Reference — Not Integrated)

⚠️ Important: The OPSX commands described below are not integrated into the AIS Project Tracker app. The prompt files live only inside the PowerCodex/ tool directory and were never copied into the tracker's .github/prompts/ folder. Attempting to use /opsx:explore or any other OPSX command in VS Code Copilot against the tracker app will not work. This section documents the workflow as designed for reference, in case a future developer decides to integrate it.

The OPSX workflow is PowerCodex's intended development lifecycle. Instead of ad-hoc Copilot chat sessions that lose context, every change would follow a structured path from idea to archived spec. All commands are intended to run inside GitHub Copilot chat in VS Code, provided the prompt files are present under .github/prompts/ in the project.

CommandPurpose
/opsx:explore <idea> Think through an idea without implementing. Use for architecture exploration, problem framing, risks, and trade-offs.
/opsx:new <change> Start a new change and inspect the first artifact instructions before drafting.
/opsx:continue <change> Continue creating or updating artifacts for an active change.

SECTION 7

How PowerCodex Was Used to Build This App

The AIS Project Tracker was built using the PowerCodex scaffold and workflow. Below is the exact sequence of steps used, which also serves as a guide for any future developer extending or rebuilding the app.

Step 1 — Scaffold the Project

# Create the PowerCodex app scaffold
npx @voyager163/powercodex ais-precision-engineering-tracker
cd ais-precision-engineering-tracker
code .

This created the Vite + React 19 + TypeScript starter, installed all dependencies, initialised OpenSpec, and set up the .github/prompts/ OPSX files ready for GitHub Copilot.

Step 2 — Initialise the Power Platform Environment

# Connect the Code App to your Power Platform environment
pac code init --environment <environmentId> --displayName "AIS Project Tracker"

This registers the app with the target environment and generates the power.config.json file that ties local development to the cloud environment.

Step 3 — Create the Dataverse Table

The crd49_AISProjectTracker1 Dataverse table was created manually in the Power Apps maker portal (make.powerapps.com) with the fields described in Section 3. The PowerCodex lifecycle engine can also automate this via portal DOM automation (see tools/lifecycle/lib/maker-recipes.js) for future tables.

Step 4 — Generate Typed Connector Hooks

After creating the Dataverse table, Power Apps tooling was used to generate typed TypeScript hooks, models, services, and validators into src/generated/. These provide fully typed useAISProjectManagerList, useCreateAISProjectManager, and useUpdateAISProjectManager hooks used throughout the app.

Step 5 — Build Features with Claude Code (Actual Method Used)

⚠️ The OPSX workflow (/opsx:propose, /opsx:apply, etc.) was not used to build this app — the prompt files were never integrated into the tracker project. Features were built directly using Claude Code with the product spec in docs/overview.md as the reference document.

Features were implemented by describing requirements to Claude Code against the docs/overview.md specification. The overview document served as the authoritative source of truth for UI behaviour, data structure, colour logic, and business rules — effectively replacing the role that OPSX artifacts would otherwise play.

For example, to build or modify the Kanban board a developer would open docs/overview.md, confirm the desired behaviour is documented there, then describe the change to Claude Code which would edit the relevant files directly (src/pages/index.tsx, src/components/kanban-card.tsx, etc.).

Step 6 — Local Development Loop

# Start the local dev server
npm run dev

# Type-check without building
npm run typecheck

# Lint
npm run lint

# Build for production
npm run build

Step 7 — Deploy to Power Platform

# Push the built app to Power Platform
pac code push

# Or for a specific environment:
pac code push --environment <environmentId>
⚠️ Authentication: The app relies entirely on Power Platform's managed authentication. Do not add a custom auth layer. Users authenticate through their Power Platform / Microsoft Entra credentials automatically.

Step 8 — Using the PowerCodex Lifecycle Dashboard (Optional)

PowerCodex ships a zero-dependency lifecycle dashboard for monitoring autonomous build loops. While the AIS Tracker was built primarily with the manual OPSX workflow above, the dashboard can be used for more autonomous iterations: 

# Serve the lifecycle dashboard
npm run lifecycle:serve          # Opens at http://localhost:4321

# Import and analyse the existing codebase
npm run lifecycle -- import --analyze

# Derive user stories from the code
npm run lifecycle -- stories

# Run the autonomous loop (real mode)
npm run lifecycle -- loop --real --app-url https://<your-app-url> --env <environmentId>

Development Guidelines to Follow

  • Read docs/overview.md before making any changes — it is the authoritative product spec
  • Keep requirements updated in docs/overview.md when behaviour changes, so the spec stays accurate
  • Use Power Platform connectors for all runtime data access; do not add a custom backend
  • Never add a custom auth layer — Power Platform handles authentication
  • Keep TypeScript strict mode on; fix all type errors before deploying
  • Do not hand-edit files under src/generated/ — regenerate them from Power Apps tooling after schema changes
  • Run npm run build and npm run typecheck before running pac code push
  • UI changes must stay consistent with Vite + React + Tailwind + shadcn/ui conventions

SECTION 10

Self-Healing Test Loop

The tracker ships a self-healing shell script that automatically runs all quality checks and — if any fail — invokes Claude Code to fix the source files, then re-runs the checks. The loop continues until everything is clean or a configurable iteration cap is reached.

ℹ️ File location: AIS Project Management Tool/apps/tracker/self-test-loop.sh

Running the Loop

cd "AIS Project Management Tool/apps/tracker"

# First time only — make it executable
chmod +x self-test-loop.sh

# Run the self-healing loop (default: up to 10 fix attempts)
./self-test-loop.sh

# Override the iteration cap
MAX_ITERATIONS=5 ./self-test-loop.sh
⚠️ Prerequisite: Claude Code must be installed. Run npm install -g @anthropic-ai/claude-code if the claude command is not on your PATH. The script will fall back to npx @anthropic-ai/claude-code automatically if the global binary is absent.

What the Loop Checks

CheckCommandWhat it catches
TypeScript npm run typecheck Type errors, missing imports, interface mismatches across all .ts / .tsx files
ESLint npm run lint Code style violations, unused variables, React hook rule violations
Build npm run build Vite bundler errors, unresolvable imports, dead code that TypeScript alone won't catch

How the Self-Heal Works

When any check fails, the script collects all error output into a single structured prompt and calls Claude Code in non-interactive print mode (claude -p). Claude is given permission to read and edit source files only (--allowedTools Read,Edit,Write) — it cannot run shell commands or delete files. After Claude finishes, the loop re-runs all three checks from scratch.

Run checks
→
Collect errors
→
Claude fixes files
→
Re-run checks
→
✅ Clean / repeat

Rules Claude Follows When Fixing

  • Edit only files under src/ unless a config file (tsconfig, vite.config, eslint) is clearly at fault
  • Do not use any as a type shortcut — fix the actual type issue
  • Do not disable ESLint rules with inline comments — fix the underlying code
  • Do not run shell commands during the fix pass — the loop handles that
  • Fix all reported errors, not just the first one

Exit Conditions

✅ Success (exit 0)

All three checks pass in the same iteration. Prints how many iterations were needed.

❌ Failure (exit 1)

Checks still fail after MAX_ITERATIONS attempts. Remaining errors are printed — manual intervention required.
